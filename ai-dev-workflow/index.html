Developer Workflow Steward

让 AI 按标准化流程开发，每一行代码都有章可循

AI 开发全流程规范为 AI Agent 提供 6 阶段门禁流水线与 15 条硬性规则， 从需求到交付，确保代码质量一致、安全可控、知识可追溯。

The Problem

AI 辅助开发 ≠ 无规范开发

没有约束的 AI 编程助手就像没有刹车的高速列车 — 速度惊人，但随时可能翻车。

  • 01 需求漂移 — AI 在长任务中逐渐偏离原始意图，交付的东西与需求无关
  • 02 安全盲区 — hardcoded secrets、SQL 注入、XSS 漏洞被 AI 像写注释一样随意生成
  • 03 质量坍塌 — 没有测试、没有 review、没有 linting，代码能跑但不可维护
  • 04 知识蒸发 — 每次对话结束，设计决策和踩坑经验随之消失，无法积累
The Solution

6 阶段门禁流水线

每个阶段都设有闸门审批 — 不通过则不能进入下一阶段。流水线即标准，标准即质量。

Hard Rules

15 条硬性规则 · 无例外

安全、质量、规范三个维度各 5 条规则。不是建议，是门禁 — 任何一条未通过，流水线冻结。

安全规则 5 条
  • S1 禁止 hardcode secret / token / password
  • S2 所有用户输入必须经过验证和清理
  • S3 第三方依赖必须审计许可证和已知漏洞
  • S4 API 端点默认要求认证，不允许匿名写操作
  • S5 敏感操作必须有审计日志
质量规则 5 条
  • Q1 新功能必须包含测试，覆盖率不低于 80%
  • Q2 变更前必须通过 lint + type-check + format
  • Q3 关键路径必须有集成测试
  • Q4 破坏性变更必须有迁移指南
  • Q5 错误信息必须可操作，不能只是 "Something went wrong"
规范规则 5 条
  • N1 每个项目必须有 README 和贡献指南
  • N2 命名遵循项目约定，不使用拼音或无意义缩写
  • N3 所有设计决策必须记录为 ADR
  • N4 Git 提交遵守 Conventional Commits
  • N5 每个阶段结束后自动归档知识到 .learnings/
Capabilities

不止是规则，是一套完整的开发操作系统

/spec
需求审讯协议
interview-me 模式：一次一个问题 + 信心指数 + 信号词检测。不等 AI 猜测你的需求，而是让它来问你。消除需求歧义。
/build
自主构建模式
/build auto — 一次批准后完全自主执行。AI 按流水线阶段逐一完成，每阶段自动门禁检查，无需反复确认。
/review
4 层 AI Code Review
Quick AI → Auto Checks → Deep AI → Human。从 lint 到架构，层层递进。前 3 层全自动，Human 只审查 AI 标记的高风险变更。
DDD
Doubt-Driven Development
5 步对抗性审查流程，在架构阶段系统性地寻找假设漏洞。不是确认你做对了什么，而是找出你做错了什么。
10x
10 大反模式清单
流程/认知/质量三维反模式，每个模式附带检测方法和修复方案。AI 在写代码时自动对照检查。
.learn
知识沉淀体系
三个层级：.learnings/（实时记录）→ patterns/（提炼模式）→ CLAUDE.md（规则固化）。每一次开发都在让下一次更高效。
Getting Started
